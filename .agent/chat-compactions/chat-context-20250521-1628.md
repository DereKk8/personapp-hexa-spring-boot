## Architectural Decisions
- **Hexagonal (Ports & Adapters)**: Three strict layers — Domain (entities + port interfaces, no framework imports), Application (use case implementations), Infrastructure (repositories, REST controllers, CLI runner, Swagger config). No extra layers or patterns.
- **Repository per aggregate**: One JPA repository per entity (Persona, Profesion, Telefono, Estudios). No generic/reusable repository abstractions.
- **Service per aggregate**: One use case class per entity, each implementing a corresponding input port interface. Use case constructor takes a specific @Qualifier output adapter.
- **Input adapter wiring**: REST/CLI adapters use `@PostConstruct` (not constructor) to instantiate the use case, because `@Autowired` fields are not available at construction time. This matches the existing pattern where `PersonaInputAdapterRest` lazily initializes in method calls.
- **cc type kept as Integer/INT**: Existing code used `Integer` throughout. Requirement said BIGINT but user chose to keep INT to avoid ~25 file refactor.
- **MongoDB left as-is**: Only `Persona` has MongoDB support (documents + repository + adapter). MongoDB documents for other entities exist but are dead code — no repositories, adapters, or endpoints reference them.
- **No pagination, no security, no extra features**: Sticking strictly to the requirements.
- **All source code in Spanish**: Variable names, class names, comments, documentation.
